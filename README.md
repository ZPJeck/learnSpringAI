# 智能商城 AI 助理平台需求说明书

## 1. 项目概述

### 1.1 项目名称

智能商城 AI 助理平台，英文名为 SmartMall AI Assistant。

### 1.2 项目背景

传统商城系统通常依赖分类导航、关键词搜索、固定 FAQ 和人工客服来解决用户购物、订单、售后等问题。随着商品数量、平台规则、活动玩法、售后政策和用户咨询量不断增加，传统方式会出现以下问题：

1. 用户难以通过关键词准确找到适合自己的商品。
2. 商品参数、活动规则、售后政策分散在多个页面或文档中。
3. 客服需要反复回答重复问题，响应效率低。
4. 用户在商品选择、订单查询、物流异常、售后申请等场景中需要更自然的交互方式。
5. 运营人员需要快速生成商品卖点、FAQ、评论摘要和活动文案。

本项目以中型商城系统为业务载体，引入 AI 问答、智能导购、知识库检索、多 Agent 协同、外部工具调用等能力，构建一个能够覆盖购物咨询、商品推荐、订单查询、售后咨询和运营辅助的智能商城平台。

### 1.3 项目定位

本项目不是单纯的商城 CRUD 系统，也不是简单的大模型聊天 Demo，而是一个面向 Spring AI 学习和实践的中型项目。项目通过真实业务场景串联 AI 应用开发中的核心能力：

1. 用户自然语言交互。
2. 多轮对话与上下文记忆。
3. 商品、订单、库存、优惠券等业务工具调用。
4. 基于知识库的 RAG 问答。
5. 多 Agent 职责拆分和任务路由。
6. 通过 MCP 标准协议接入外部业务能力。
7. AI 调用日志、质量评估、安全防护和可观测性。

### 1.4 项目目标

#### 1.4.1 业务目标

1. 构建一个可运行的智能商城原型。
2. 支持用户浏览商品、查询商品、创建订单、查询订单。
3. 提供 AI 智能导购能力，根据预算、偏好、使用场景推荐商品。
4. 提供 AI 商品问答能力，回答商品参数、适用场景、对比建议等问题。
5. 提供 AI 客服能力，回答退换货、发票、保修、物流等问题。
6. 提供 AI 订单助手能力，查询订单状态、物流进度和售后状态。
7. 提供 AI 运营辅助能力，生成商品卖点、FAQ、评论摘要和活动文案。
8. 支持知识库文档管理、向量化、检索和答案来源追踪。

#### 1.4.2 学习目标

1. 深入理解 Spring AI 在业务系统中的使用方式。
2. 熟练使用 ChatClient 构建同步和流式对话接口。
3. 掌握 Prompt、System Prompt、User Prompt 和结构化输出设计。
4. 掌握 Tool Calling，把业务服务暴露给模型调用。
5. 掌握 RAG 完整流程，包括文档解析、文本切分、Embedding、向量入库和相似度检索。
6. 掌握 VectorStore 的使用和向量库调参。
7. 掌握 Advisor 在问答增强、记忆增强和检索增强中的使用。
8. 掌握 MCP Client 与 MCP Server 的基本开发方式。
9. 掌握多 Agent 的职责拆分、路由、上下文传递和结果汇总。
10. 建立 AI 应用工程化意识，包括权限、安全、日志、评估、降级和监控。

## 2. 用户角色

### 2.1 消费者

消费者使用商城前台完成商品浏览、智能咨询、商品对比、下单、订单查询、物流查询和售后咨询。

### 2.2 客服人员

客服人员使用 AI 助手快速查询平台政策、总结用户问题、生成回复建议，并在复杂问题中辅助人工处理。

### 2.3 运营人员

运营人员负责商品资料维护、知识库文档上传、活动规则维护、商品卖点生成、FAQ 生成和评论摘要分析。

### 2.4 系统管理员

系统管理员负责用户权限、AI 模型配置、Agent 配置、MCP 工具状态、系统日志和异常监控。

## 3. 项目范围

### 3.1 包含范围

1. 商城基础业务：商品、分类、SKU、库存、订单、优惠券、评论、售后。
2. 用户端 AI 对话：商品推荐、商品问答、订单查询、售后咨询、优惠咨询。
3. 管理端 AI 能力：知识库管理、文档向量化、运营内容生成、调用日志查看。
4. RAG 知识库：文档上传、解析、切片、向量化、检索、来源追踪。
5. 多 Agent 编排：意图识别、Agent 路由、专用 Agent 执行和结果汇总。
6. Tool Calling：商品、订单、库存、优惠券、物流、售后等工具调用。
7. MCP 集成：模拟订单、物流、优惠三个外部 MCP 服务。
8. AI 可观测性：模型调用日志、工具调用日志、Agent 执行轨迹、RAG 命中记录。

### 3.2 不包含范围

1. 不接入真实支付渠道，支付使用模拟支付。
2. 不接入真实物流服务，物流使用模拟 MCP 服务。
3. 不实现完整企业级电商营销系统，只实现优惠券和满减规则的基础版本。
4. 不实现复杂机器学习推荐算法，推荐主要通过规则筛选、结构化检索和大模型推理完成。
5. 不追求高并发秒杀能力，重点关注 AI 与业务系统集成。
6. 不实现完整客服工单流转系统，只提供简化工单创建和查询。

## 4. 业务场景

### 4.1 智能导购

用户通过自然语言描述预算、偏好、用途和限制条件，系统推荐合适商品。

示例问题：

1. 我预算 500 元，想买一款适合跑步的蓝牙耳机，最好防水、续航久。
2. 推荐一款适合办公室使用的机械键盘。
3. 3000 元以内，有没有适合学生党的轻薄笔记本？

系统应完成：

1. 识别用户购买意图。
2. 提取预算、品类、场景、品牌、功能偏好等条件。
3. 查询商品库和库存。
4. 结合商品说明、评论摘要、售后政策生成推荐理由。
5. 返回候选商品列表和对比建议。
6. 支持用户继续追问和缩小范围。

### 4.2 商品知识问答

用户围绕某个商品提问，系统基于商品详情、说明书、FAQ 和售后政策回答。

示例问题：

1. X200 耳机支持多设备连接吗？
2. 这台咖啡机可以用胶囊咖啡吗？
3. 这个商品保修多久？

系统应完成：

1. 识别商品名称、型号或商品 ID。
2. 检索商品详情和知识库。
3. 基于检索证据回答。
4. 展示答案来源。
5. 未命中明确依据时提示无法确认。

### 4.3 订单查询

用户查询订单状态、发货状态、取消条件和物流进度。

示例问题：

1. 帮我查一下订单 202606230001 到哪里了。
2. 我的订单为什么还没发货？
3. 这个订单还能取消吗？

系统应完成：

1. 识别订单查询意图。
2. 校验订单归属。
3. 查询订单状态。
4. 查询物流进度。
5. 给出当前状态和下一步建议。

### 4.4 售后客服

用户咨询退货、换货、维修、退款、发票和保修等问题。

示例问题：

1. 商品用了 5 天坏了，可以退吗？
2. 耳机拆封后还能七天无理由吗？
3. 发票可以重开吗？

系统应完成：

1. 判断售后问题类型。
2. 检索售后政策和商品类目规则。
3. 在用户提供订单号时查询订单状态。
4. 判断是否满足退换货或维修条件。
5. 给出可执行建议。
6. 必要时创建客服工单。

### 4.5 优惠活动咨询

用户咨询优惠券、满减、活动叠加和最优价格。

示例问题：

1. 这个商品有没有优惠券？
2. 帮我算一下买两件最便宜多少钱。
3. 满减和优惠券可以叠加吗？

系统应完成：

1. 查询商品活动规则。
2. 查询用户可用优惠券。
3. 计算最优价格。
4. 解释优惠构成。
5. 对不可叠加规则给出说明。

### 4.6 运营辅助

运营人员使用 AI 生成和优化商品内容。

示例任务：

1. 根据商品参数生成 5 个卖点。
2. 根据评论总结用户最满意和最不满意的点。
3. 根据商品说明生成 FAQ。
4. 根据活动规则生成客服话术。

系统应完成：

1. 读取商品资料和评论数据。
2. 生成结构化内容。
3. 标注内容来源。
4. 支持人工编辑后保存。

## 5. 功能需求

### 5.1 用户与权限

1. 用户可以注册、登录和退出。
2. 用户角色至少包括消费者、客服、运营、管理员。
3. 消费者只能查询自己的订单和售后单。
4. 运营人员可以管理商品资料和知识库。
5. 管理员可以查看 AI 调用日志和系统配置。

### 5.2 商品模块

1. 支持商品新增、修改、删除、查询、上下架。
2. 支持商品分类管理。
3. 支持 SKU 管理。
4. 支持商品属性维护。
5. 支持商品标签维护。
6. 支持商品图文描述维护。
7. 支持按关键词、分类、品牌、价格区间、标签检索商品。
8. 支持商品详情页展示 AI 推荐理由和常见问答。

### 5.3 库存模块

1. 支持 SKU 库存查询。
2. 支持下单时库存扣减。
3. 支持取消订单时库存恢复。
4. 支持低库存提醒。
5. AI 回答库存问题时必须调用库存工具或库存接口。

### 5.4 订单模块

1. 支持创建订单。
2. 支持查询订单详情。
3. 支持取消订单。
4. 支持模拟支付。
5. 支持订单状态流转。
6. 支持发货状态更新。
7. 支持用户通过 AI 查询订单状态。

### 5.5 优惠券模块

1. 支持优惠券创建和启停。
2. 支持用户领取优惠券。
3. 支持查询用户可用优惠券。
4. 支持优惠金额计算。
5. 支持 AI 解释优惠构成。

### 5.6 评论模块

1. 支持用户评价商品。
2. 支持商品评分统计。
3. 支持 AI 总结好评点和差评点。
4. 支持 AI 生成评论摘要。

### 5.7 售后模块

1. 支持用户创建售后申请。
2. 支持查询售后单状态。
3. 支持售后类型：退货、换货、维修、退款。
4. 支持客服处理售后单。
5. 支持 AI 根据政策和订单状态给出售后建议。

### 5.8 AI 对话模块

1. 支持普通对话。
2. 支持流式输出。
3. 支持多轮会话。
4. 支持会话列表和消息历史查询。
5. 支持会话标题自动生成。
6. 支持识别用户意图。
7. 支持输出结构化结果。
8. 支持 AI 回复中展示工具调用结果摘要。
9. 支持 AI 回复中展示知识来源。
10. 支持用户对 AI 回复进行反馈。

### 5.9 RAG 知识库模块

1. 支持上传 Markdown、TXT、PDF、DOCX、XLSX 等知识文档。
2. 支持文档元数据维护，包括文档类型、适用分类、关联商品、版本号和生效状态。
3. 支持文档解析。
4. 支持文本切片。
5. 支持生成 Embedding。
6. 支持写入向量库。
7. 支持相似度检索。
8. 支持按商品、分类、品牌、文档类型过滤。
9. 支持知识库重建。
10. 支持知识下线和版本管理。
11. 支持 RAG 检索测试。
12. 支持查看每次 AI 回答命中的知识片段。

### 5.10 多 Agent 模块

系统至少包含以下 Agent：

1. Router Agent：负责识别意图并路由。
2. Product Advisor Agent：负责商品推荐和商品对比。
3. Knowledge QA Agent：负责知识库问答。
4. Order Agent：负责订单和物流查询。
5. After-Sales Agent：负责售后咨询。
6. Promotion Agent：负责优惠活动咨询。
7. Operation Agent：负责运营内容生成。

每个 Agent 应具备：

1. 独立职责说明。
2. 独立 System Prompt。
3. 独立可用工具集合。
4. 独立知识库检索范围。
5. 独立输出格式。
6. 执行日志记录。

### 5.11 Tool Calling 模块

系统至少提供以下工具：

1. 商品搜索工具。
2. 商品详情工具。
3. 商品对比工具。
4. 库存查询工具。
5. 用户优惠券查询工具。
6. 最优价格计算工具。
7. 订单状态查询工具。
8. 物流轨迹查询工具。
9. 售后政策查询工具。
10. 客服工单创建工具。

工具调用要求：

1. 参数必须结构化。
2. 入参必须校验。
3. 调用必须记录日志。
4. 高风险操作必须二次确认。
5. 工具返回结果应精简，避免返回无关数据。
6. 模型不得绕过业务服务直接修改核心数据。

### 5.12 MCP 模块

系统应包含以下 MCP 服务：

1. Order MCP Server：提供订单状态和售后状态能力。
2. Logistics MCP Server：提供物流轨迹和预计送达能力。
3. Promotion MCP Server：提供优惠计算和活动规则能力。

主商城应用应作为 MCP Client：

1. 能连接多个 MCP Server。
2. 能获取 MCP 工具清单。
3. 能把 MCP 工具注册给对应 Agent。
4. 能记录 MCP 工具调用日志。
5. 能在 MCP 服务不可用时降级处理。

### 5.13 AI 日志与可观测性

1. 记录每次 AI 请求的用户、会话、输入、输出摘要。
2. 记录模型名称、调用耗时、Token 用量。
3. 记录 Agent 路由结果。
4. 记录工具调用名称、参数摘要、结果摘要和耗时。
5. 记录 RAG 命中文档和片段。
6. 记录异常信息和降级原因。
7. 管理端支持按用户、会话、Agent、工具、时间范围查询日志。

## 6. 非功能需求

### 6.1 性能需求

1. 普通业务接口平均响应时间小于 500ms。
2. RAG 检索平均耗时小于 1s。
3. 流式聊天首 token 响应时间小于 3s。
4. 普通 AI 问答整体响应时间应尽量控制在 10s 内。
5. 文档向量化任务可异步执行。

### 6.2 准确性需求

1. 商品价格、库存、订单状态必须来自数据库或业务工具。
2. 售后政策、活动规则、保修政策必须来自知识库或业务配置。
3. AI 不得虚构商品参数、售后承诺、订单状态。
4. 当知识库未命中时，AI 应明确说明无法确认。
5. AI 回复应优先给出可执行建议。

### 6.3 安全需求

1. 用户只能访问自己的订单、优惠券和售后单。
2. 管理端接口必须鉴权。
3. 工具调用必须进行权限校验。
4. 取消订单、创建售后、创建工单等操作必须有业务校验。
5. 日志中应避免保存完整手机号、地址、身份证等敏感信息。
6. 需要防范 Prompt Injection，知识库内容不得覆盖系统安全规则。

### 6.4 可维护性需求

1. Prompt 模板应独立维护。
2. Agent 配置应独立维护。
3. 工具定义应模块化。
4. 知识库文档应支持版本管理。
5. 模型供应商应支持切换。
6. 向量库应支持替换。

### 6.5 可观测性需求

1. 业务日志、AI 日志、工具日志、MCP 日志应能关联同一次请求。
2. 系统应支持错误码和异常追踪。
3. 管理端应支持查看模型调用和工具调用统计。
4. 需要保留关键 AI 链路的 Trace 信息。

## 7. 验收标准

### 7.1 基础商城验收

1. 可以维护商品、分类、SKU 和库存。
2. 可以创建订单、支付订单、取消订单和查询订单。
3. 可以领取优惠券并计算优惠价格。
4. 可以发布评论并查看评论摘要。

### 7.2 AI 问答验收

1. 用户可以进行多轮 AI 对话。
2. 聊天接口支持普通响应和流式响应。
3. 系统能识别商品推荐、商品问答、订单查询、售后咨询、优惠咨询等常见意图。
4. 价格、库存、订单、物流类问题必须触发工具调用。
5. 政策、说明书、FAQ 类问题必须触发 RAG 检索。

### 7.3 RAG 验收

1. 上传文档后可以完成解析、切片和向量化。
2. 用户问题可以命中相关知识片段。
3. AI 回答可以展示来源。
4. 知识不足时 AI 不编造答案。
5. 管理端可以查看知识命中日志。

### 7.4 多 Agent 验收

1. Router Agent 能正确分发常见问题。
2. 不同 Agent 使用不同工具和知识范围。
3. 管理端可以查看 Agent 执行轨迹。
4. Agent 之间可以传递必要上下文。

### 7.5 MCP 验收

1. 主应用可以连接多个 MCP Server。
2. AI 可以通过 MCP 查询物流、优惠和订单信息。
3. MCP 工具调用有日志。
4. MCP 服务不可用时有降级提示。

### 7.6 工程验收

1. 项目可以通过本地配置启动。
2. 关键业务接口有测试。

## 8. 实施阶段

### 8.1 第一阶段：商城基础骨架

完成用户、商品、分类、SKU、库存、订单、优惠券、评论、售后等基础模块。

### 8.2 第二阶段：AI 聊天接入

完成模型配置、普通聊天、流式聊天、会话管理和基础 Prompt。

### 8.3 第三阶段：Tool Calling

完成商品搜索、库存查询、订单查询、优惠计算等工具，并接入 AI 对话。

### 8.4 第四阶段：RAG 知识库

完成文档上传、解析、切片、Embedding、向量入库、相似度检索和来源展示。

### 8.5 第五阶段：多 Agent 编排

完成 Router Agent 和多个业务 Agent，支持意图路由、工具隔离和执行轨迹。

### 8.6 第六阶段：MCP 集成

完成订单、物流、优惠 MCP Server，并在主应用中作为 MCP Client 调用。

### 8.7 第七阶段：质量评估与优化

完成测试问题集、RAG 命中率评估、工具调用准确率评估、Prompt 优化和降级策略。

## 9. 测试问题集

### 9.1 商品推荐

1. 我预算 500 元，想买跑步耳机，推荐几款。
2. 帮我找一款适合办公室用的机械键盘。
3. 这两款手机哪个更适合拍照？

### 9.2 商品问答

1. X200 耳机支持防水吗？
2. A300 咖啡机可以用胶囊咖啡吗？
3. 这台笔记本支持扩展内存吗？

### 9.3 订单物流

1. 帮我查订单 202606230001 到哪里了。
2. 我的订单为什么还没发货？
3. 这个订单还能取消吗？

### 9.4 售后政策

1. 商品用了 5 天坏了，可以退吗？
2. 耳机拆封后还能七天无理由吗？
3. 发票可以重开吗？

### 9.5 优惠活动

1. 这个商品有没有优惠券？
2. 帮我算一下买两件最便宜多少钱。
3. 满减和优惠券可以叠加吗？

### 9.6 运营辅助

1. 帮我给这款耳机生成 5 个商品卖点。
2. 根据这些评论总结用户最满意和最不满意的点。
3. 为这款咖啡机生成 FAQ。
